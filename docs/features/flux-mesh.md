# Flux Mesh — LAN-Local Node Discovery & Routing

Flux is NAVIG's zero-config, LAN-only peer mesh. Multiple NAVIG daemons on the
same subnet discover each other via UDP multicast, share load information, and
can forward chat/AI requests to the best available peer when the local GPU or
model context is busy.

---

## Architecture

```
Node A (you)                 Node B (peer LAN)
┌─────────────────┐          ┌─────────────────┐
│  navig daemon   │◄────────►│  navig daemon   │
│  port 8789      │ UDP 5354 │  port 8789      │
│  NodeRegistry   │          │  NodeRegistry   │
│  MeshDiscovery  │          │  MeshDiscovery  │
└─────────────────┘          └─────────────────┘
         ▲                            ▲
         │ HTTP /mesh/*               │
     navig-bridge                  navig-bridge
     (VS Code)                    (VS Code)
```

- **Transport**: UDP multicast `224.0.0.251:5354` — LAN only, never WAN.
- **Node ID**: `navig-{os}-{hostname_slug}-{mac4}` — stable across DHCP rebinds.
- **Auth**: Cross-node HTTP calls use `gateway.mesh_token` (auto-generated,
  stored in `~/.navig/config.yaml`).
- **Backend priority**: local daemon → best mesh peer → Copilot (last resort).

---

## Configuration

All options live under `~/.navig/config.yaml`:

```yaml
gateway:
  mesh_token: "<auto-generated hex32>"   # written on first start

mesh:
  enabled: true       # set false to disable discovery entirely
  # multicast_group and port can be overridden if your LAN blocks 224.0.0.251
```

The mesh starts automatically when `navig gateway start` (or `navig service
start`) is run, provided `mesh.enabled` is not `false`.

## CLI Commands

### `navig flux status`

Show own Flux node identity and a health summary of known peers.

```
navig flux status [--json]
```

### `navig flux peers`

List all known peers and their health/load.

```
navig flux peers [--json] [--health online|degraded|offline]
```

### `navig flux ping <url>`

Manually bootstrap a peer by pointing at its gateway URL.
Useful when UDP multicast is blocked or for cross-subnet WireGuard setups.

```
navig flux ping http://10.0.0.42:8789
navig flux ping http://10.8.0.2:8789 --json
```

### `navig flux route <message>`

Route a chat/AI message to the best available peer.
Falls back to local daemon if no peer is suitable.

```
navig flux route "summarise this for me"
navig flux route "run inference" --capability gpu
navig flux route "ping pong" --node navig-linux-server
navig flux route "hello" --json
```

## REST API

The gateway exposes three mesh endpoints (always on the local daemon):

| Method | Path           | Description |
|--------|----------------|-------------|
| `GET`  | `/mesh/peers`  | Returns self + all known peers as JSON. |
| `POST` | `/mesh/ping`   | Bootstraps a peer: `{"url": "http://..."}` |
| `POST` | `/mesh/route`  | Routes a request: `{"message": "...", "capability": "...", "target_node_id": "..."}` |

## VS Code (navig-bridge)

The **Flux Nodes** panel in the NAVIG sidebar (broadcast icon) shows live peer
health, gateway URL, and load. Commands accessible from the panel:

| Command                  | Action |
|--------------------------|--------|
| `navig.mesh.setTarget`   | Pin a specific peer as routing target |
| `navig.mesh.clearTarget` | Remove pinned target, resume auto-routing |
| `navig.mesh.addNode`     | Manually add peer by URL |
| `navig.mesh.refresh`     | Force panel refresh |
| `navig.mesh.scanLan`     | Trigger active LAN scan |

---

## Health States

| State      | Meaning |
|------------|---------|
| `online`   | Last seen < 90 s ago |
| `degraded` | Last seen 90–300 s ago |
| `offline`  | Last seen > 300 s ago |

Peers absent for > 1800 s are evicted from the registry automatically.
