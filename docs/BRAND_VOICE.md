# NAVIG Brand Voice

Use this when writing public-facing NAVIG copy.

## Default posture

NAVIG is terminal-first, operator-focused, remote-friendly, automation-ready, and AI-assisted without pretending the operator is optional.

Write like that.

- Direct
- Controlled
- Specific
- Low-hype
- Slightly severe
- Honest about limits

## What NAVIG is

- An infrastructure CLI and runtime
- Built for operators, not tourists
- Useful from the terminal first
- Serious about security, provenance, and explicit approval
- Experimental and still moving
- Solo-built and accountable

## What NAVIG is not

Do not make NAVIG sound like:

- a happy SaaS mascot
- a generic AI wrapper
- a dashboard company pretending terminals are obsolete
- a lifestyle productivity app
- a fake enterprise brand padded with empty language

## Tone rules

### Use short declarative sentences

Lead with what it does. Then why it matters. Then the boundary.

Example:

> NAVIG is an infrastructure CLI and runtime for managing remote systems from one place. It reduces SSH sprawl, repeated manual work, and brittle operator habits. It assists with AI where useful, but keeps the human in control.

### State limits plainly

Use the real words:

- experimental
- not yet stable
- may change between releases
- AI assists, does not replace operator control

### Authority comes from detail

Prefer:

- concrete verbs
- operational detail
- test, build, and security language
- visible constraints

Do not reach for inflated adjectives.

## Lore usage

Lore is an accent, not the operating language.

Words like `Kraken`, `Sentinel`, `sigil`, `Leviathan`, `Deepwatch`, `abyss`, and `signal` can appear in headers, launch copy, visuals, and identity flourishes.

Do not let them take over README copy, policy docs, or product explanations.

## Voice split

### Founder voice

Use `I` when talking about:

- why NAVIG exists
- tradeoffs
- roadmap priorities
- what is not done yet
- funding and independence

### Product voice

Use `NAVIG` when talking about:

- features
- commands
- releases
- policies
- security handling
- official build requirements

### Community voice

Use `the project` or `the community` for:

- contributions
- discussions
- support channels
- forks

## Copy formulas

### Homepage / README

`[Hard claim]. [Operational pain]. [Practical outcome]. [Constraint or honesty line].`

Example:

> Run your infrastructure from one place. Less SSH sprawl, fewer repeated commands, less fragile glue. NAVIG helps you inspect, operate, and automate across hosts without hiding the machine behind a toy dashboard. Experimental, but built to be useful now.

### Release notes

`[What changed]. [What broke before]. [What is safer, faster, or clearer now].`

### Founder post

`[I built or fixed X]. [Why it mattered operationally]. [What still is not done].`
